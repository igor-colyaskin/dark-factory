## Ориентир

Довести DF до уровня, когда она сможет генерировать любые приложения
в рамках заданного бюджета, законности, доступности ресурсов.

Это **не цель**, а ориентир. Цель — навыки: научиться работать с командой
агентов, понять как устроен диалог человека и LLM, получить опыт
проектирования agent-систем.

Если v0.1 — велосипед, v0.2 — скутер, то v100.500 — самолёт.
Самолёт построить не получится (и не нужно — уже летают). Но каждый шаг
по маршруту оправдывает себя полученным опытом.

---

## Главный вектор: от workflow к agency

**Workflow (сейчас):** пользователь говорит → агенты выполняют → готово.
Агенты — руки без головы.

**Agency (цель):** агенты говорят "подожди" вместо "готово".
Спрашивают, предупреждают, проверяют, помнят.

Переход происходит в три стадии:
NEGOTIATE → VERIFY → REMEMBER v0.3 v0.4 v0.5


Каждая стадия делает следующую мощнее:
- Negotiate без Verify = спросил правильно, сделал криво, не заметил
- Verify без Negotiate = проверил, но проверял не то
- Remember без Negotiate+Verify = помнит ошибки, но не умеет их не повторять

## Выпущенные версии

### v0.1 — Велосипед ✅
Pipeline работает. Агенты вызываются через OpenRouter. Результат — код
в workspace/. Учебная задача решена.

### v0.2 — Скутер ✅
Приложения деплоятся на Fly.io. Публичная ссылка, QR-код.
Получатель не ставит Docker/Node — достаточно браузера.

### v0.2.1 — Шасси ✅
Products page: архив всех созданных приложений. CRUD через UI.
App slug из текста заказа (первый микро-шаг agency).

### v0.3 — NEGOTIATE ✅
Архитектор останавливает конвейер и разговаривает с заказчиком.
Два режима: clarify (структурированные вопросы с вариантами ответов)
и spec (полная спецификация перед разработкой). Spec Review UI
с кнопками Start/Cancel. Developer prompt принимает spec напрямую.

Метрики dogfooding:
- Ясный заказ (Pomodoro): $0.12, 1 мин, 0 раундов вопросов
- Неоднозначный заказ (Habits): $0.18, 1:28, 1 раунд вопросов

## Следующие версии

---

### v0.4 — VERIFY

**Тема:** агент проверяет свою работу и отчитывается.

**A. Verification Report**
После деплоя агент-верификатор сверяет spec с результатом. Выдаёт
структурированный отчёт: что реализовано, что нет, почему.

**B. Feedback Loop**
Пользователь видит отчёт и говорит: "кнопки должны быть синими,
а не красными". DF запускает точечную доработку без полного пересоздания.

**Pipeline после v0.4:**
Order → Clarify → Spec → Dev → Deploy → Verify → Feedback → Fix → Done


**Ключевое обучение:** как агент оценивает собственную работу?
Как структурировать feedback, чтобы он был actionable?

---

### v0.5 — REMEMBER

**Тема:** агент учится на прошлом опыте.

**A. Order by Reference**
"Как #25, но кнопки синие" — архитектор находит прошлый заказ, извлекает
spec, применяет дифф.

**B. Pattern Library**
Архитектор накапливает базу знаний о типовых заказах. "TODO-приложение"
→ известный паттерн с проверенной архитектурой.

**Ключевое обучение:** как агенту дать память? Сколько контекста — 
достаточно, а сколько — шум?

---

### v0.6+ — CAPABILITIES (на зрелой agency-базе)

Расширение набора создаваемых приложений. Появляется **только когда
agency-слой достаточно зрел**, чтобы новые capabilities не ломали
качество.

Возможные направления:
- Telegram-каналы с автопубликацией
- Headless-сервисы (schedulers, scrapers)
- Multi-page приложения
- Внешние API-интеграции (с управлением secrets)

---

### v1.0 — горизонт

DF становится **осмысленным собеседником**: ведёт переговоры, предлагает
альтернативы, оценивает стоимость, согласует spec, отчитывается.

Не дата, а **состояние зрелости agency-слоя**.

## Открытый список идей

Не привязаны к конкретным версиям. Всплывут, когда придёт время,

## Принципы (напоминание)

1. **Agency over capabilities** — глубина переговоров важнее ширины умений
2. **Маленькие шаги ради понимания** — понял, прежде чем двигаться дальше
3. **Guided decisions over free-form** — варианты, а не пустое поле
4. **Не копим хвосты** — устаревшее удаляем
