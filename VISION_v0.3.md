Краткое содержание
Dark Factory v0.1 и v0.2 создают простые веб-страницы: страница открыта — что-то работает, страница закрыта — ничего не происходит. Это учебный класс приложений.

v0.3 предлагает фундаментальный сдвиг: DF учится генерировать автономные сервисы — приложения, которые продолжают работать без пользователя на странице. Первый и флагманский класс таких сервисов — Telegram-каналы с автопубликацией.

Мотивация владельца:

освоить новый инженерный класс задач (secrets management, schedulers, headless apps, интеграции с внешними сервисами);
получить реально полезный инструмент — возможность за 10 минут запустить собственный Telegram-канал со своей идеей.

1. Что меняется концептуально
Текущая парадигма DF (v0.1–v0.2)

Пользователь → DF → Публичная страница
                     ↓
           Пользователь открыл → работает
           Пользователь закрыл → ничего не происходит

Пример: TODO-список. Данные в памяти. Без пользователя — сервер работает вхолостую.

Новая парадигма DF (v0.3+)

Пользователь → DF → Автономный сервис
                     ↓
           Работает сам по себе
           Пользователь иногда проверяет результат
           Результат доставляется через внешний канал
           (Telegram, email, push — в будущем)

4. Что принципиально нового нужно построить в DF
4.1. Secrets Management
Проблема: Telegram-боту нужен TELEGRAM_BOT_TOKEN. Приложению с LLM нужен OPENROUTER_API_KEY. Этот токен нельзя коммитить в workspace, нельзя отдавать в публичный URL, нельзя вшивать в код.

Решение:

Architect-агент в своём ответе указывает needs_secrets: [{name, description, how_to_get}].
Между ARCH_REVIEW и DEV_WORKING появляется новое состояние оркестратора: AWAITING_SECRETS.
UI DF показывает пользователю форму: "Приложение требует такие-то ключи. Вот инструкция, как их получить. Введи сюда."
Fly Manager расширяется методом setSecrets(appName, {KEY: VALUE, ...}) — это fly secrets set перед fly deploy.
Developer-агент генерирует код, который читает ключи из process.env.*, никогда не хардкодит.
AC Checker проверяет: нет ли в коде hardcoded токенов (регулярки на типичные паттерны).
4.2. Schedulers
Проблема: пост "Блюдо дня" должен публиковаться каждое утро. Приложение должно уметь выполнять задачи по расписанию.

Решение для v0.3 (простое): setInterval внутри Node.js процесса. Для раз-в-день задач достаточно с запасом. Никакой отдельный cron-инфраструктуры не нужен.

Побочный эффект: auto_stop_machines = stop в fly.toml становится несовместим со schedulers. Если машина спит — setInterval не тикает.

Решение: для приложений с needs_scheduler: true в fly.toml ставится min_machines_running = 1 и auto_stop_machines = false. Это увеличивает стоимость приложения с ~$0 до ~$2–3/мес. Это осознанный трейд-офф, пользователь должен быть предупреждён в UI.

4.3. Headless apps
Проблема: Telegram-каналу не нужен browser UI. Но деплой на Fly.io требует health endpoint.

Решение:

Приложение всегда имеет минимальный Express-сервер с роутом GET / → возвращает "OK" или статус-страницу (uptime, последний пост, etc.).
Блок PICKUP в UI DF меняется для headless-приложений: вместо "открой публичный URL и играйся" — "вот ссылка на твой Telegram-канал [@channel], первый пост через N минут. Вот служебный URL (если нужно посмотреть статус)".
4.4. External integrations
Проблема: приложение обращается к Telegram API, возможно к OpenRouter, возможно к другим API.

Решение для v0.3: не пытаться построить "универсальный интегратор". Вместо этого — каталог поддерживаемых интеграций (whitelist):

Telegram Bot API (приоритет 1, демо-заказ).
OpenRouter (для LLM внутри приложений).
Далее — по мере появления заказов.
Для каждой интеграции:

Памятка для Architect: "если пользователь просит X, используй Y".
Инструкция для пользователя DF: как получить ключ.
Шаблон кода для Developer.
Проверки для AC Checker.

5. Какие вопросы ещё открыты
Эти вопросы надо обсудить перед началом разработки (после "переночевать"):

5.1. Кто платит за LLM внутри сгенерированных приложений?
Предварительное решение owner'а: Вариант 1 — пользователь DF даёт свой OpenRouter-ключ, приложение его использует как свой secret. Риск утечки минимизируется тем, что стадия "кащейная" — DF используется только самим владельцем.

5.2. Нужен ли отдельный оркестратор для каждого типа headless-приложения?
Пока гипотеза: нет, достаточно одного расширенного pipeline, который узнаёт тип приложения из ответа Architect и ведёт себя соответственно (показывает или скрывает блок secrets, выбирает правильный шаблон для Developer, корректирует блок PICKUP).

5.3. Как DF помогает пользователю создать бота/канал в Telegram?
Этот шаг технически не может быть автоматизирован (BotFather требует ручных действий в Telegram). Нужна подробная пошаговая инструкция в UI DF, возможно со скриншотами. Рассматривать как часть UX v0.3.

5.4. Что делать при отказе в деплое приложения со scheduler?
Если приложение не стартовало на Fly — текущий оркестратор делает retry деплоя. Но для headless-приложения "деплой прошёл, но планировщик не тикает" — это тоже отказ, и его сложнее обнаружить. Возможно, нужен новый вид AC: "после деплоя подожди N минут и проверь, что в логах есть ожидаемые события".

5.5. Как очищать headless-приложения?
Сейчас TTL = 24 часа hard. Для Telegram-канала это абсурд — он должен жить долго. Либо TTL для headless-приложений отключается, либо становится продлеваемым (в UI DF появляется панель "Мои приложения" раньше, чем планировалось — v0.2.1 приходится делать перед v0.3).

7. Принципы, которых придерживаемся
Эволюция, не революция. Ничего из v0.2 не выкидываем. Добавляем поверх.
Dogfooding first. Каждая новая версия DF оправдана только если owner сам начинает ею пользоваться.
Кащейная стадия сохраняется. Всё для одного пользователя (owner'а). Никакого мультипользовательского режима.
Wishlist документируется, но не делается наперёд. Если в процессе v0.3 возникнет идея "а ещё хорошо бы X" — X идёт в v0.4+ секцию roadmap, не в текущий scope.
Стоимость прозрачна. Headless-приложения дороже CRUD (не засыпают). Пользователь видит это заранее.

Приложение A. Глоссарий
Headless app — приложение без постоянного browser UI. Работает само, пользователь приходит за результатом в другое место (Telegram, email).
Scheduler — компонент, выполняющий задачи по расписанию. В v0.3 — простой setInterval.
Secret — конфиденциальная переменная окружения (токен API, пароль). Передаётся через Fly secrets, никогда не попадает в workspace или git.
Dogfooding — практика, когда создатели продукта сами им пользуются. Главный источник обратной связи.
Demo-заказ — конкретный пример приложения, на который ориентируется разработка текущей версии DF (аналог "smoke test" для всего pipeline).
