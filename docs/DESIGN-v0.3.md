# DESIGN-v0.3.md — NEGOTIATE

## Решение Игоря

### Flow
Order → Architect → [Questions?] → [More questions?] → [Blocker?] → Summary → Start


- Architect всегда завершает на Summary (даже если вопросов не было)
- Max 3 раунда вопросов (защита от бесконечного цикла)
- ~5 вопросов за раунд (мягкий лимит в промпте, не в коде)
- Blocker = hard stop, один вопрос, ждём решения
- Между раундами — текст-progress от архитектора, не "Раунд N/M"

### Кнопки Summary

- **Start** — запуск разработки
- **Cancel** — возврат в IDLE без расходов
- **Поправить** — disabled placeholder (активируется в v0.4)

## Решение Claude
Ядро: negotiate как цикл разговора

round = 0, history = []

loop:
  response = callArchitect(order, history)
  
  switch response.mode:
    "clarify"  → показать вопросы, ждать ответов
                  history.push({ questions, answers })
                  round++, continue loop
    
    "blocker"  → показать стоп-сигнал, ждать решения
                  если cancel → IDLE
                  если продолжить → history.push({ blocker, decision })
                  continue loop
    
    "spec"     → показать Summary, ждать Start/Cancel
                  break loop

  if round >= 3 → принудить spec на следующем вызове

  Оркестратор крутит этот цикл. Каждая итерация — один вызов архитектора + одно ожидание пользователя. Между ними — SSE обновления для UI.

  Что меняется, что остаётся
Не трогаем (всё от DEV_WORKING и дальше)

DEV_WORKING → DEV_CHECK → TEST_RUNNING → DELIVERING → DEPLOYING → DONE

File manager, AC checker, fly manager, products, tabs, cost tracker — всё как есть.

Самое сложное: промпт архитектора
Это 80% успеха всей фазы. Всё остальное — обвязка.

Промпт должен надёжно делать три вещи:

1. Решать: спрашивать или нет

Правило: спрашивай ТОЛЬКО если ответ меняет АРХИТЕКТУРУ.
- "Данные в памяти или в файле?" → меняет архитектуру → спрашивай
- "Кнопки синие или зелёные?" → не меняет → не спрашивай
- "Нужна ли авторизация?" → меняет → спрашивай
- "Какой шрифт?" → не меняет → не спрашивай

2. Формулировать хорошие вопросы

Плохо: "Какие функции вы хотите?"  (слишком открыто, ступор)
Хорошо: "Что делать с выполненными задачами?"
        • Вычёркивать в списке
        • Перемещать в архив
        • Удалять автоматически
        • Другое

3. Знать когда остановиться

Правило: минимум вопросов для старта. 
Если сомневаешься — не спрашивай, сделай разумное допущение.
Детали отшлифуются в feedback (v0.4).

Моя стратегия для промпта
Не один гигантский system prompt, а два шаблона user prompt:

Шаблон A (первый вызов):
  "Вот заказ: {order}. 
   Проанализируй. Если нужны уточнения — задай вопросы.
   Если всё ясно — выдай spec."

Шаблон B (повторный вызов):
  "Вот заказ: {order}.
   История уточнений: {history}.
   Текущий раунд: {round} из {max}.
   Если нужно ещё уточнить — задай вопросы.
   Если достаточно — выдай spec."

System prompt один и тот же — роль, формат, примеры, ограничения.

questionsLog: кто собирает?
В дизайне сказано: архитектор включает questionsLog в spec. Но я бы не доверял это LLM. Надёжнее:

Оркестратор ведёт clarifyHistory (факт, не интерпретация)
Когда архитектор возвращает spec → оркестратор сам приклеивает историю
На Summary пользователь видит точные вопросы и свои точные ответы
Это убирает риск, что архитектор "перефразирует" или потеряет часть Q&A.

Главный риск
Архитектор может непредсказуемо переключаться между режимами. Например:

Ясный заказ → всё равно задаёт вопросы (назойливо)
Неясный заказ → сразу выдаёт spec (рискованно)
Возвращает невалидный JSON
Митигация:

Few-shot примеры в промпте (2-3 примера каждого режима)
Валидация ответа: если нет поля mode → retry
Если mode: "clarify" но questions пустой → трактовать как spec
Фаза 1 целиком посвящена итерации промпта — 5+ тестовых вызовов
Порядок реализации (почему такой)

Phase 1: Prompt          ← сначала заставить LLM работать правильно
Phase 2: Mock            ← чтобы тестировать UI без расходов на API
Phase 3: Orchestrator    ← state machine под новый flow
Phase 4: UI Clarifying   ← первый визуальный результат
Phase 5: UI Summary      ← второй визуальный результат
Phase 6: Integration     ← всё вместе
Phase 7: Release         ← доки

Prompt первый, потому что если архитектор не может надёжно выдавать нужный формат — всё остальное бессмысленно. Лучше узнать это на фазе 1, чем на фазе 6.
