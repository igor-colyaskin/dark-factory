# TRACKER v0.2.1 — My Apps

## Обзор

v0.2.1 — тихая инфраструктурная версия DF. Вводит персистентное хранилище
сведений о созданных приложениях и UI для их просмотра/удаления.

**Метафора:** каркас самолёта получает шасси.

**Философская нагрузка:** Products — не просто список, а **архив решений**
Dark Factory. В будущих версиях (v0.3+) карточка приложения обогатится
spec'ом, verification report'ом и другими артефактами agency-слоя.

## Принятые решения

### Функциональность
- Страница Products с карточками всех успешно созданных приложений.
- Табы Order / Products в шапке, переключение через client-side toggle.
- Карточка: number (#506), id (df-abc123), дата, стоимость, время, URL,
  кнопки "Детали" и "Стереть".
- Details — inline-expand с сырым architectOutput.
- Удаление через modal с подтверждением, удаляет и с Fly.io, и из архива.
- Пустое состояние с приглашением перейти в Order.

### Хранилище
- Файл `state/apps.json` в корне проекта.
- Структура: `{ version: 1, nextNumber: N, apps: [...] }`.
- Атомарная запись (tmp + rename).
- `/state/` добавлен в `.gitignore`.
- Номера монотонные, не переиспользуются при удалении.

### Интеграция
- Запись в архив — в момент успешного DONE (вариант B).
- Существующие приложения (созданные до v0.2.1) не импортируются.
- Контекст Architect'а в v0.2.1 не трогаем (stateless остаётся).

### Не-цели (явно)
- Нет поиска/фильтрации приложений.
- Нет live health-check (реальной проверки "живо или нет").
- Нет скриншотов/preview.
- Нет ссылок на прошлые заказы из текста (это v0.2.2).
- Нет персистентного хранения неудачных попыток.
- Нет master-detail layout.
- Нет глубоких URL-роутов.

### Структура записи в apps.json

```json

  "id": "df-abc123",
  "flyAppName": "df-abc123",
  "number": 506,
```

## Фазы

### Фаза 1. Хранилище ⬜

**Цель:** работающий модуль `server/apps-store.js` + файл `state/apps.json`.

**Что делаем:**
1. Добавить `/state/` в `.gitignore`.
2. Создать `server/apps-store.js` с функциями:
   - `init()` — проверка/создание `state/apps.json`, обработка битого файла
     (бэкап как `apps.json.corrupt-<timestamp>` + создание нового пустого).
   - `addApp(appData)` — добавление записи, автоприсвоение number из
     nextNumber, инкремент nextNumber.
   - `getAllApps()` — все записи, отсортированные по createdAt DESC.
   - `getApp(id)` — одна запись по id.
   - `deleteApp(id)` — удаление из архива.
3. Атомарная запись через tmp + rename.
4. `init()` вызывается при старте сервера.
5. Начальное содержимое: `{ "version": 1, "nextNumber": 1, "apps": [] }`.

**Проверка:**
- Запустить сервер, убедиться что `state/apps.json` создался пустой.
- Через Node REPL или тестовый скрипт — вызвать addApp 2-3 раза,
  проверить корректность номеров.
- Удалить вручную через deleteApp — getAllApps не возвращает.
- Испортить apps.json (вставить мусор), перезапустить — бэкап создан,
  файл новый пустой.

**Коммит:** `v0.2.1 phase 1: apps-store module`

**Оценка:** 1 сессия с Roo.

### Фаза 2. Интеграция с orchestrator ⬜

**Цель:** успешно завершённые заказы попадают в архив.

**Что делаем:**
1. В `orchestrator.js` в точке перехода DEPLOYING → DONE добавить вызов
   `appsStore.addApp(...)`.
2. Собрать данные для записи: `session.order`, `session.architectOutput`,
   URL от FlyManager, метрики.
3. Обработка ошибки записи: логируем, **но не проваливаем заказ**. DONE
   наступает в любом случае, если приложение задеплоено на Fly.

**Проверка:**
- Сделать один реальный заказ (production или mock-full).
- Убедиться, что в `apps.json` появилась запись с правильными полями.
- Искусственно "сломать" запись (сделать `apps.json` read-only) —
  заказ всё равно завершается успешно, warning в логах.

**Коммит:** `v0.2.1 phase 2: orchestrator integration`

**Оценка:** 0.5 сессии.

### Фаза 3. API endpoints ⬜

**Цель:** клиент может читать список и удалять записи.

**Что делаем:**
1. Добавить в server (routing):
   - `GET /api/my-apps` → `appsStore.getAllApps()`.
   - `GET /api/my-apps/:id` → `appsStore.getApp(id)` или 404.
   - `DELETE /api/my-apps/:id` → FlyManager.destroyApp + appsStore.deleteApp.
2. Последовательность удаления и обработка edge cases:
   - Fly отвечает "app not found" → считаем удалённым, идём дальше.
   - Fly отвечает ошибкой сети → возвращаем ошибку клиенту, запись в
     apps.json **не удаляем**.
   - Fly отвечает "access denied" → возвращаем ошибку клиенту, запись
     в apps.json **не удаляем**.
3. Добавить `destroyApp(appName)` в FlyManager, если его ещё нет
   (`fly apps destroy <name> --yes`).

**Проверка:**
- curl или browser DevTools: GET /api/my-apps возвращает список.
- GET /api/my-apps/:id возвращает одну запись или 404.
- DELETE /api/my-apps/:id — убивает приложение на Fly и запись в архиве.
- DELETE для несуществующего id → 404.

**Коммит:** `v0.2.1 phase 3: API endpoints`

**Оценка:** 0.5-1 сессия.

### Фаза 4. UI: табы Order / Products ⬜

**Цель:** в шапке видны две табы, переключаются, работающий pipeline не
ломается.

**Что делаем:**
1. В HTML добавить шапку с двумя кнопками-табами + иконка самолёта справа.
2. В `app.js` — client-side toggle: показываем `<div id="page-order">`
   или `<div id="page-products">` через display: none.
3. Весь существующий UI Order'а оборачивается в `page-order`.
4. Создаётся `page-products` с заглушкой ("Products page — coming").
5. Активная таба визуально выделена (underline или background).

**Важно:** переключение **не затрагивает** state machine оркестратора.
Можно быть в DEV_WORKING на Order, переключиться на Products, вернуться —
pipeline продолжает идти.

**Проверка:**
- Запустить заказ, переключиться на Products в процессе, вернуться — 
  pipeline идёт, состояние видно.
- Страница Products показывает заглушку.
- Активная таба визуально выделена.

**Коммит:** `v0.2.1 phase 4: tabs UI`

**Оценка:** 1 сессия.

### Фаза 5. UI: страница Products — список ⬜

**Цель:** страница Products показывает карточки приложений.

**Что делаем:**
1. При переходе на таб Products — запрос `GET /api/my-apps`.
2. Рендер списка карточек. Одна карточка = один app.
3. Пустое состояние:
   ```
   Пока ничего нет.
   Перейди в [Order] и сделай свой первый заказ.
   ```
4. Анатомия карточки (без активных действий в этой фазе):
   - `#506 | df-abc123` — крупно, вверху карточки.
   - Первые ~80 символов заказа.
   - Дата создания (форматированная), стоимость ($0.14), время (2m 40s).
   - Ссылка "🌐 https://... [Открыть] [Скопировать]".
   - Кнопки "Детали" и "Стереть" — visible, но placeholder (без логики).
5. Обновление списка: при переключении на таб + после успешного
   завершения нового заказа (если пользователь был на Order).

**Проверка:**
- Создать 2-3 заказа, они появляются в Products.
- Пустой список — показывается приглашение перейти в Order.
- Карточка отображает все поля корректно.
- Формат даты/стоимости/времени читаемый.
- После нового заказа — карточка появляется без перезагрузки страницы.

**Коммит:** `v0.2.1 phase 5: Products page list`

**Оценка:** 1 сессия.

### Фаза 6. UI: Details (inline-expand) ⬜

**Цель:** кнопка "Детали" раскрывает architectOutput под карточкой.

**Что делаем:**
1. Клик на "Детали" → expand блока с `<pre>{architectOutput}</pre>`.
2. Повторный клик — collapse.
3. Кнопка меняет подпись: "Детали" ↔ "Скрыть".
4. Длинный architectOutput — scroll внутри блока, не ломаем вёрстку.

**Проверка:**
- Expand/collapse работает.
- Несколько карточек можно раскрыть одновременно (не мешают друг другу).
- Длинный architectOutput не ломает вёрстку.

**Коммит:** `v0.2.1 phase 6: card details`

**Оценка:** 0.3 сессии.

### Фаза 7. UI: удаление с подтверждением ⬜

**Цель:** полный путь удаления через модальное окно.

**Что делаем:**
1. Клик на "Стереть" → открывается модалка:
   ```
   Удалить приложение #506 (df-abc123)?
   
   Это:
     • удалит приложение с Fly.io (оно перестанет работать)
     • удалит запись из архива
   
   Отменить нельзя.
   
                           [Отмена]  [Да, удалить]
   ```
2. Клик "Да, удалить" → DELETE /api/my-apps/:id.
3. В процессе удаления — кнопка disabled, показывается "Удаление...".
4. Успех → модалка закрывается, карточка исчезает из UI.
5. Ошибка → показываем сообщение ("Не удалось удалить: [причина]"),
   модалка не закрывается, можно попробовать ещё раз или отменить.
6. Клик "Отмена" или вне модалки → закрытие без удаления.

**Проверка:**
- Полный цикл: создал заказ → удалил через UI → проверил что и в Fly
  dashboard нет, и в Products нет, и в apps.json нет.
- Отмена не удаляет.
- Если Fly вернул ошибку (отключить интернет) — UI корректно обрабатывает,
  запись в архиве сохраняется.

**Коммит:** `v0.2.1 phase 7: delete with confirmation`

**Оценка:** 1 сессия.
