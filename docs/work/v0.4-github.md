# v0.4 — GitHub Integration

**Цель версии:** каждое сгенерированное приложение попадает в GitHub
как отдельный репозиторий в аккаунте пользователя. 24-часовой TTL
на Fly перестаёт быть проблемой потери — код живёт вечно. Открывается
путь к REMEMBER (v0.5), где Architect будет работать с существующим
кодом, а не регенерировать его.

**Pipeline после v0.4:**
`Order → Negotiate → Spec → Start → Dev → Test → Deploy → GitHub Commit → Done`

---

## Дизайн-решения

### Model

- **OAuth App** для авторизации (не PAT)
- **Один репозиторий на приложение**
- **Имя:** `df-<slug>-<number>` (e.g. `df-todo-app-509`) — одно имя
  и для Fly app, и для GitHub repo. Унификация упрощает отладку
  и навигацию между компонентами.
- **Privacy:** приватные по умолчанию (настраивается в Settings)
- **Owner:** аккаунт пользователя (не DF)

### Содержимое репозитория

Стандартная структура:

\`\`\`
<repo-name>/
  README.md       ← автогенерация: описание, запуск, бэдж DF
  SPEC.md         ← копия spec от Architect
  app.js          ← код от Developer
  package.json
  Dockerfile
  fly.toml
  public/         ← если есть frontend
  .gitignore      ← стандартный Node.js
\`\`\`

### Timing

Коммит происходит **после успешного деплоя**. Если деплой упал —
репозиторий не создаётся. Концепция: репозиторий содержит только
работающий код.

### Deploy flow

Без изменений. Деплой по-прежнему из `workspace/` через flyctl.
GitHub — архив и история, не источник деплоя.

### Fly app name (изменение)

Fly app names до v0.4: `df-<slug>` (e.g. `df-todo-app`).
Fly app names с v0.4: `df-<slug>-<number>` (e.g. `df-todo-app-509`).

Старые приложения не переименовываются. Изменение применяется
к новым заказам.

### GitHub Topics (автоматически)

- `dark-factory`
- `ai-generated`
- `nodejs`
- тематические из spec (опционально, решим в фазе)

### README.md — автогенерация

Шаблон примерно такой:

\`\`\`markdown
# <App Name>

> Generated by [Dark Factory](https://github.com/user/dark-factory) · Order #509

<Краткое описание из spec.summary>

## Features
<список из spec.features>

## Running

\`\`\`bash
npm install
npm start
\`\`\`

## About

Generated on <date> from order: "<original order text>"

See [SPEC.md](./SPEC.md) for full specification.
\`\`\`

## Открытые вопросы

### 1. OAuth App: в чьём аккаунте регистрировать?

Чтобы OAuth работал, нужно зарегистрировать OAuth App на GitHub.
У неё есть owner — пользователь или организация.

- [ ] A. Зарегистрировать в личном GitHub-аккаунте (твой)
- [ ] B. Создать organization "Dark Factory" и зарегистрировать там
- [ ] C. Личный сейчас, переведём в organization позже

**Решение:** C — личный GitHub-аккаунт. Перенос в organization
опционально в будущем (поддерживается GitHub без потери OAuth).

### 2. Scope токена

Какие права запрашивать при OAuth?

- [ ] A. `repo` (полный доступ к репозиториям пользователя)
- [ ] B. `public_repo` (только публичные) — но мы хотим приватные
- [ ] C. Fine-grained token через GitHub App (современнее, но сложнее)

**Решение:** A — scope `repo` (полный доступ к репозиториям).
Необходим для создания приватных репо.

### 3. Хранение токена

Где DF хранит OAuth-токен после авторизации?

- [ ] A. `state/github-tokens.json` (простой файл, как apps.json)
- [ ] B. Отдельный файл на пользователя (если будут мульти-юзер)
- [ ] C. Зашифрованное хранение (добавить encryption layer)

**Решение:** A — `state/github-tokens.json`. Плоский файл
consistent с `apps.json`. Файл в gitignore. Encryption
отложено до момента, когда появится мульти-юзер или
публичный deployment.

### 4. Что делать при отсутствии подключения GitHub?

Пользователь не подключил GitHub. Заказал приложение. Что делать?

- [ ] A. Заблокировать заказ: "Подключите GitHub, чтобы продолжить"
- [ ] B. Выполнить заказ, предупредить: "GitHub не подключен,
      приложение без репозитория"
- [ ] C. Сохранить код локально как fallback (см. Сценарий 2
      из предыдущего обсуждения), показать предупреждение

**Решение:** B — выполнить заказ, предупредить пользователя.
В apps.json пишется `githubUrl: null`. В UI при получении
приложения показывается notice "GitHub не подключен — код
не сохранён. Подключите в Settings для будущих заказов".

### 5. SPEC.md формат

Копировать spec от Architect'а в markdown. Формат?

- [ ] A. JSON-dump в fenced code block (просто, читабельно)
- [ ] B. Красивый markdown (headers, списки) — нужна сериализация
- [ ] C. И JSON, и читабельный markdown

**Решение:** отложено до Phase 5. Рабочая гипотеза — вариант C
(и JSON, и markdown). Финализируется при реализации на основе
реального вида spec.

### 6. Что показываем в Products UI?

Есть GitHub-репо для каждого приложения. Как это отражается в UI?

- [ ] A. Иконка GitHub рядом с карточкой, клик → github.com
- [ ] B. Секция "Source Code" в Details с ссылкой
- [ ] C. Embedded preview (showcasing README.md внутри DF)

**Решение:** A+B — иконка GitHub на карточке (клик → github.com
в новой вкладке) и секция "Source Code" в Details с ссылкой и
кнопкой "Delete repository". Delete — с подтверждением,
удаляет только репо, приложение на Fly остаётся.
README внутри DF не показываем — проще и тоньше.

### 7. Fly app name vs GitHub repo name

Fly app name сейчас: `df-<slug>`. Repo name предлагается: `<slug>-<number>`.
Они разные. Это нормально или нужно унифицировать?

- [ ] A. Оставить разными (Fly name имеет `df-` префикс, repo — номер)
- [ ] B. Унифицировать: `df-<slug>` для обоих
- [ ] C. Добавить номер в Fly name тоже: `df-<slug>-<number>`

**Решение:** C — унифицированное имя `df-<slug>-<number>`
для Fly app и GitHub repo. Префикс `df-` остаётся (читается
как Dark Factory signature), номер гарантирует уникальность.
Подробности — в секции "Дизайн-решения".

### 8. Метрики в apps.json

Сейчас apps.json хранит базовые поля. Добавлять ли `githubUrl`?

- [ ] A. Да, новое поле `githubUrl` в записи
- [ ] B. Новое вложенное поле `github: { url, repoName, createdAt }`
- [ ] C. Пересчитывать URL по шаблону `github.com/<user>/<repo>`

**Решение:** отложено до Phase 3. Рабочая гипотеза — вариант A
(плоское поле `githubUrl`). Финализируется при реализации.

### 9. Что, если репо с таким именем уже существует?

При `<slug>-<number>` коллизия маловероятна (номер уникален), но:

- [ ] A. Падать с ошибкой
- [ ] B. Добавлять суффикс (`todo-app-509-a`)
- [ ] C. Использовать alternative-имя от Architect'а (нужно
      попросить Architect генерировать 2 варианта?)

**Решение:** отложено до Phase 2. Рабочая гипотеза — вариант B
(суффикс `-2`, `-3`, ...). Финализируется при реализации
обработки API-ответов от GitHub.

## План реализации

### Phase 0: Design ✅

Дизайн-решения зафиксированы выше. Открытые вопросы закрываются
до начала Phase 1.

### Phase 1: OAuth App Setup

**Цель:** GitHub OAuth App зарегистрирована, callback работает,
токен успешно получается и сохраняется.

**Подфазы:**

**1.1** Регистрация OAuth App на GitHub:
- Имя, homepage URL, callback URL
- Получить Client ID, Client Secret
- Записать в `.env`

**1.2** UI: Settings page с кнопкой "Connect GitHub"
- Клик → редирект на GitHub OAuth authorize URL
- После авторизации GitHub вернёт `code` на callback

**1.3** Backend: OAuth callback handler
- Обмен `code` на access token (POST на GitHub token endpoint)
- Сохранение токена в `state/github-tokens.json` (или выбранный вариант)
- Редирект пользователя на Settings с success

**1.4** UI: отобразить статус подключения
- "✓ Connected as @username" или "Not connected"
- Кнопка "Disconnect"

**1.5** Тестирование flow: подключение, переподключение, отключение.

**Критерий выхода:** Ты можешь подключить GitHub-аккаунт через UI DF,
токен сохраняется, отображается username.

**Коммит:** `v0.4 phase 1: github oauth setup`

### Phase 2: GitHub Client Module

**Цель:** отдельный модуль `server/github-client.js`, который умеет
создавать репозитории и коммитить файлы через GitHub API.

- [ ] 2.1 Модуль `github-client.js`: класс GitHubClient с токеном
- [ ] 2.2 Метод `createRepo(name, { private, description, topics })`
- [ ] 2.3 Метод `commitFiles(repoName, files, commitMessage)`
- [ ] 2.4 Метод `getUser()` — для проверки токена
- [ ] 2.5 Обработка ошибок: rate limits, expired tokens, name conflicts
- [ ] 2.6 Тест модуля на твоём реальном токене
- [ ] 2.7 Коммит: `v0.4 phase 2: github client module`

### Phase 3: Orchestrator Integration

**Цель:** после успешного деплоя orchestrator вызывает GitHub Client
и создаёт репозиторий с кодом.

- [ ] 3.1 Новый state `COMMITTING_TO_GITHUB` между DEPLOYING и DONE
- [ ] 3.2 Переход DEPLOYING → COMMITTING_TO_GITHUB (при успешном деплое)
- [ ] 3.3 Переход COMMITTING_TO_GITHUB → DONE
- [ ] 3.4 Обработка ошибок: если GitHub упал — логируем, но идём в DONE
      (деплой успешен, репо — бонус)
- [ ] 3.5 Сохранение `githubUrl` в apps.json
- [ ] 3.6 SSE broadcast: progress "Committing to GitHub..."
- [ ] 3.7 Коммит: `v0.4 phase 3: orchestrator integration`

## Риски v0.4

| Риск | Митигация |
|------|-----------|
| OAuth callback сложнее, чем ожидалось (HTTPS requirements, redirect URL issues) | Фаза 1 — отдельная. Если заблокирует — временно fallback на PAT, довести OAuth как полировку |
| GitHub API rate limits (5000/час для authenticated) | На одного пользователя × пара приложений в день = далеко от лимита. Мониторить, если станет проблемой — добавить retry с backoff |
| Токен истёк / отозван пользователем | Детектировать 401 при API-вызовах, снова просить authorize |
| Репо создался, но коммит упал (частичный failure) | Транзакционность: создаём репо → сразу первый коммит. Если коммит упал — удаляем репо (откат) |
| README.md выглядит плохо для нестандартных spec | Визуальный тест на 5-6 разных приложениях до релиза |
