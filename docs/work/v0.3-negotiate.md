# v0.3 — NEGOTIATE

**Цель версии:** архитектор останавливает конвейер и разговаривает
с заказчиком. Спрашивает, предупреждает, показывает spec. Первый
настоящий agency-шаг.

**Pipeline до v0.3:**
`Order → Architect → Approve → Dev → Test → Deploy → Done`

**Pipeline после v0.3:**
`Order → Analyze → [Clarify?] → [Blocker?] → Summary → Start → Dev → Test → Deploy → Done`

---

## Дизайн-решения

### Flow

`Order → Architect → [Questions?] → [More questions?] → [Blocker?] → Summary → Start`

- Architect всегда завершает на Summary (даже если вопросов не было)
- Max 3 раунда вопросов (защита от бесконечного цикла)
- ~5 вопросов за раунд (мягкий лимит в промпте, не в коде)
- Blocker = hard stop, один вопрос, ждём решения
- Между раундами — текст-progress от архитектора, не "Раунд N/M"

### Формат ответа архитектора

Три режима: `clarify`, `blocker`, `spec`.

#### Mode: clarify
```json
{
  "mode": "clarify",
  "questions": [
    {
      "id": "q1",
      "text": "Как должны храниться данные задач?",
      "options": [
        "В памяти сервера (проще, пропадут при перезапуске)",
        "В файле на сервере (переживут перезапуск)",
        "В базе данных (надёжно, сложнее)"
      ],
      "allowOther": true
    }
  ],
  "progress": "Ещё пара вопросов и можно начинать."
}
```

#### Mode: blocker
```json
{
  "mode": "blocker",
  "message": "Для отображения прогноза погоды нужен внешний API.",
  "options": [
    "OpenWeatherMap — бесплатный план (1000 запросов/день), нужна регистрация",
    "Open-Meteo — полностью бесплатный, без ключа, но менее точный"
  ],
  "allowOther": true
}
```

#### Mode: spec
```json
{
  "mode": "spec",
  "appSlug": "todo-app",
  "spec": {
    "summary": "Простое TODO-приложение с хранением в памяти",
    "features": [
      "Добавление задач",
      "Отметка выполненных",
      "Удаление задач"
    ],
    "screens": [
      "Одна страница: список задач + форма добавления"
    ],
    "constraints": [
      "Node.js + Express + статический фронтенд",
      "Данные в памяти сервера"
    ],
    "warnings": [
      "⚠ Данные не сохраняются при перезапуске сервера"
    ],
    "estimatedCost": "\$0.30–0.50",
    "estimatedTime": "2–3 мин"
  },
  "questionsLog": [
    { "question": "Как хранить данные?", "answer": "В памяти сервера" }
  ]
}
```

### UI: страница вопросов

- Radio-кнопки для options (не dropdown — всё видно сразу)

### Кнопки Summary

- **Start** — запуск разработки
- **Cancel** — возврат в IDLE без расходов
- **Поправить** — disabled placeholder (активируется в v0.4)

### Orchestrator states (v0.3)

IDLE
  → ARCH_WORKING (user submits order)
  
ARCH_WORKING
  → CLARIFYING   (architect returns mode: "clarify")
  → BLOCKER      (architect returns mode: "blocker")
  → SPEC_REVIEW  (architect returns mode: "spec")
  
CLARIFYING
  → ARCH_WORKING (user answers, re-call architect with context)
  
BLOCKER
  → ARCH_WORKING (user chooses to continue)
  → IDLE         (user chooses to cancel)
  
SPEC_REVIEW
  → DEV_WORKING  (user clicks Start)
  → IDLE         (user clicks Cancel)


### Архитектурные замечания

- `questionsLog` собирается оркестратором из `clarifyHistory`,
  не архитектором (надёжность)
- Developer получает spec целиком как контекст (вместо старого
  `architectOutput`)
- Старый формат `architectOutput` (files + summary) заменяется на spec
- Архитектор совмещает роли Architect + Planner (спецификация +
  оценка). Разделение — в v0.5+. См. `docs/contracts.md` → Architect.

## План реализации

### Phase 0: Design ✅ (эта фаза)

Дизайн-решения зафиксированы выше. Открытые вопросы закрываются
до начала Phase 1.

### Phase 1: Architect Prompt v2

**Цель:** архитектор надёжно работает в трёх режимах (clarify/blocker/spec).
Это 80% успеха v0.3 — если промпт не работает, всё остальное бесполезно.

**Подфазы:**

**1a.** Переписать system prompt:
- Роль: "tech lead на первой встрече с заказчиком"
- JSON schema для трёх режимов с примерами
- Правило: спрашивай ТОЛЬКО если ответ меняет архитектуру
- Правило: минимум вопросов для старта, детали — в feedback (v0.4)
- Правило: формулируй вопросы, которые подталкивают к ответам
- Запрет: не спрашивай про цвета, шрифты, анимации

**1b.** Два шаблона user prompt:
- Template A (первый вызов): order
- Template B (повторный): order + clarifyHistory + round/maxRounds

**1c.** Калибровочный тест — 10 эталонных заказов:
- 3× ясный заказ (TODO, calculator, clock) → ожидается `mode: "spec"` сразу
- 3× неоднозначный заказ ("приложение для работы", "простое что-нибудь") → `mode: "clarify"`
- 2× заказ с внешней зависимостью ("покажи погоду", "новости") → `mode: "blocker"`
- 2× повторный вызов с history → `mode: "spec"`

**Критерий выхода:** 9 из 10 заказов дают ожидаемый режим с валидным JSON.

**Коммит:** `v0.3 phase 1: architect prompt v2`

## Риски v0.3

| Риск | Митигация |
|------|-----------|
| Architect "сваливается" в один из режимов (только clarify или только spec) | Калибровочный тест из Phase 1c. Критерий выхода: 9/10 |
| LLM путает clarify и blocker | Если проявится — слить в один режим (см. открытый вопрос 2) |
